Preface

Abstract

The Alumni Portal is a full-stack web application designed to help educational institutions maintain a structured, searchable database of their graduates. Built on a MySQL relational database backend and a React.js frontend, the system enables alumni registration, profile management, peer messaging, referral requests, and administrator-controlled data maintenance.

The project demonstrates core DBMS concepts including Entity–Relationship modelling, schema design, normalization to 3NF, DDL/DML/DQL operations, aggregate functions, multi-table JOINs, views, and triggers — all integrated into a production-style REST API served by Node.js and Express.

An embedded Analytics dashboard surfaces live SQL query results — company-wise distribution, graduation-year histograms, INNER JOIN and LEFT JOIN result sets — directly in the browser, satisfying the testing and output requirements of the project specification.

Section 01

Introduction

Educational institutions maintain thousands of alumni records that are often stored in isolated spreadsheets, email threads, or outdated paper registers. This makes it difficult to search, contact, or analyse graduate data. A centralised, database-driven portal solves these problems by offering structured storage, real-time search, and role-based access control.

The Alumni Portal addresses this gap. It provides a student-facing interface for browsing alumni profiles and sending peer messages, and an admin-facing interface for adding, editing, and deleting records. All data mutations are automatically recorded in an audit log via a MySQL TRIGGER, and a VIEW surfaces a denormalised summary of alumni for quick reporting.

This report maps every stage of the project — from problem definition and ER modelling through SQL implementation and frontend integration — to the professor's 11-step checklist.

  • Expose all database operations via a RESTful API (Node.js + Express + mysql2).

  • Implement role-based access control (admin vs user) enforced on both frontend and backend.
  • Implement peer messaging and referral request functionality.
  • Surface live SQL analytics (aggregates, JOINs) on an Analytics dashboard tab.

Section 06

Normalization (up to 3NF)

Each table in the schema has been analysed and verified to satisfy 1NF, 2NF, and 3NF (and BCNF).

1NF — First Normal Form ✔
Rule: All column values are atomic; no repeating groups; every row has a unique primary key.

Satisfied because: Every column stores a single, indivisible value (e.g., year is a single integer, company is a single string). No column holds a list or array. Each row is identified uniquely by its auto-increment id (or log_id).
2NF — Second Normal Form ✔
Rule: Already in 1NF and all non-key attributes are fully functionally dependent on the entire primary key (no partial dependencies).

Satisfied because: Every table uses a single-column surrogate primary key (id INT AUTO_INCREMENT). A partial dependency can only arise when the PK is composite — since our PKs are all single-column, partial dependencies are structurally impossible. Therefore 2NF is automatically satisfied.
3NF — Third Normal Form ✔
Rule: Already in 2NF and there are no transitive dependencies (no non-key column determines another non-key column).

Example check — alumni table:
  • id → name, id → email, id → year, id → company ✔
  • company does not determine year, and year does not determine company.
  • No non-key attribute determines any other non-key attribute → no transitive dependency → 3NF satisfied.
Example check — messages table:
  • sender does not determine receiver or message.
  • All attributes depend only on the primary key id → 3NF satisfied.
BCNF — Boyce–Codd Normal Form ✔
Rule: For every non-trivial functional dependency X → Y, X must be a superkey.

Satisfied because: The only determinant in every table is the primary key column. There are no candidate keys other than id (except email UNIQUE — but email → other columns means email is itself a superkey, which is valid in BCNF).

SQL Queries

DDL — Data Definition Language

CREATE TABLE users

CREATE TABLE IF NOT EXISTS users (
  id         INT AUTO_INCREMENT PRIMARY KEY,
  name       VARCHAR(255) NOT NULL,
  email      VARCHAR(255) NOT NULL UNIQUE,
  password   VARCHAR(255) NOT NULL,
  role       ENUM('user','admin') DEFAULT 'user',
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE alumni

CREATE TABLE IF NOT EXISTS alumni (
  id      INT AUTO_INCREMENT PRIMARY KEY,
  name    VARCHAR(255) NOT NULL,
  email   VARCHAR(255) NOT NULL UNIQUE,
  year    INT NOT NULL,
  company VARCHAR(255) NOT NULL
);

CREATE TABLE alumni_audit_log

CREATE TABLE IF NOT EXISTS alumni_audit_log (
  log_id      INT AUTO_INCREMENT PRIMARY KEY,
  alumni_id   INT,
  alumni_name VARCHAR(255),
  action      VARCHAR(50),
  logged_at   TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE messages

CREATE TABLE IF NOT EXISTS messages (
  id          INT AUTO_INCREMENT PRIMARY KEY,
  sender      VARCHAR(255) NOT NULL,
  receiver    VARCHAR(255) NOT NULL,
  message     TEXT NOT NULL,
  is_referral BOOLEAN DEFAULT FALSE,
  sent_at     TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

DQL — SELECT Queries & Aggregates

Basic SELECT — All alumni

SELECT * FROM alumni ORDER BY id DESC;

WHERE — Filter by company

SELECT * FROM alumni WHERE company = 'Google';

BETWEEN — Year range filter

SELECT name, email, year, company
FROM alumni
WHERE year BETWEEN 2019 AND 2022
ORDER BY year ASC;

COUNT — Total alumni

SELECT COUNT(*) AS total_alumni FROM alumni;

GROUP BY — Alumni per company

SELECT company, COUNT(*) AS total
FROM alumni
GROUP BY company
ORDER BY total DESC;

GROUP BY — Alumni per graduation year

SELECT year, COUNT(*) AS graduates
FROM alumni
GROUP BY year
ORDER BY year DESC;

Aggregate Functions — MAX / MIN / AVG

SELECT
  MAX(year) AS latest_batch,
  MIN(year) AS earliest_batch,
  ROUND(AVG(year), 0) AS avg_graduation_year
FROM alumni;

HAVING — Companies with more than 1 alumnus

SELECT company, COUNT(*) AS cnt
FROM alumni
GROUP BY company
HAVING cnt > 1;

INNER JOIN — Alumni with audit log

SELECT
  a.name       AS alumni_name,
  a.company,
  a.year,
  l.action,
  l.logged_at
FROM alumni a
INNER JOIN alumni_audit_log l ON a.id = l.alumni_id
ORDER BY l.logged_at DESC;

LEFT JOIN — All alumni (with or without audit entry)

SELECT
  a.name       AS alumni_name,
  a.year,
  a.company,
  COALESCE(l.action, 'No log') AS audit_action
FROM alumni a
LEFT JOIN alumni_audit_log l ON a.id = l.alumni_id
ORDER BY a.year DESC;

Sub-query — Latest batch alumni

SELECT * FROM alumni
WHERE year = (SELECT MAX(year) FROM alumni);

View & Triggers

VIEW — alumni_summary

CREATE OR REPLACE VIEW alumni_summary AS
SELECT
  id, name, email,
  year   AS graduation_year,
  company,
  CONCAT('Class of ', year, ' | ', company) AS summary
FROM alumni
ORDER BY year DESC;

-- Query the view:
SELECT * FROM alumni_summary;

TRIGGER 1 — Log every INSERT on alumni

CREATE TRIGGER after_alumni_insert
AFTER INSERT ON alumni
FOR EACH ROW
BEGIN
  INSERT INTO alumni_audit_log (alumni_id, alumni_name, action)
  VALUES (NEW.id, NEW.name, 'INSERT');
END;

TRIGGER 2 — Log every DELETE on alumni

CREATE TRIGGER after_alumni_delete
AFTER DELETE ON alumni
FOR EACH ROW
BEGIN
  INSERT INTO alumni_audit_log (alumni_id, alumni_name, action)
  VALUES (OLD.id, OLD.name, 'DELETE');
END;

Section 11

Conclusion

The Alumni Portal successfully demonstrates a complete database application development lifecycle. Starting from a real-world problem — the lack of structured alumni management in educational institutions — the project progresses through formal ER modelling, normalised schema design, comprehensive SQL query implementation, and a polished full-stack web application.

All four DBMS concepts mandated by the project specification have been implemented: a VIEW (alumni_summary) for denormalised reporting, two TRIGGERs (after_alumni_insert and after_alumni_delete) for automated audit logging, JOINs (INNER and LEFT JOIN) connecting the alumni and audit log tables, and aggregate functions (COUNT, MAX, MIN, AVG, GROUP BY, HAVING) surfaced live in the Analytics dashboard.

The project showcases how a relational database can power a production-quality web application with role-based access control, real-time search, peer messaging, and comprehensive data analytics — all driven by well-structured SQL queries exposed through a REST API.
